# /challenge - 方案挑战指令

## 触发方式
用户说 `/challenge` 或 "帮我挑战这个方案"、"找找这个设计的问题"
可以附带文档路径：`/challenge 01_游戏设计/系统设计/气盾机制.md`
也可以直接在对话中粘贴方案内容后说 `/challenge`

## 执行流程

### 1. 获取待挑战的方案
- 如果用户指定了文件路径 → 读取该文件
- 如果用户粘贴了内容 → 直接使用对话中的内容
- 如果都没有 → 询问用户"你想让我挑战哪个方案？"

### 2. 加载相关背景（可选但推荐）
搜索知识库中与该方案相关的文档，作为挑战依据：
- 竞品如何解决同类问题（`02_参考资料/`）
- 以往迭代中遇到过哪些类似问题（`05_复盘总结/`）
- 历史会议中是否有相关讨论（`03_会议纪要/`）

### 3. 从五个维度发起挑战
按以下结构输出挑战意见：

```
⚔️ 【方案挑战报告】

🎯 方案核心理解：
{用一两句话复述你理解的方案核心，确认没有误读}

---

❶ 逻辑自洽性
{方案内部是否存在矛盾？规则A和规则B会不会打架？}

❷ 玩家体验预判
{玩家实际操作时会遇到什么困惑？学习成本如何？}

❸ 极端情况 / 边界漏洞
{有没有可以被滥用的极端操作？边界情况处理了吗？}

❹ 实现复杂度风险
{哪些部分程序实现难度高？有没有更简单的替代方案？}

❺ 与现有系统的冲突
{和知识库中其他系统文档相比，有没有联动冲突？}

---

💡 总结建议：
{最需要优先解决的 1~2 个问题，以及修改方向}
```

### 4. 态度原则
- **只提问题，不替用户做决策**
- 每个挑战点给出具体的场景或例子，而不是泛泛而谈
- 如果方案整体没有明显漏洞，如实说明，不要为了挑战而挑战
- 结尾给出优先级排序，帮助用户聚焦最重要的问题

## 补充规则
- 挑战的目的是帮助完善方案，语气保持建设性而非否定性
- 如果涉及数值问题，可以提出极端情况下的数值测试用例
- 每条挑战意见尽量配一个"如果...那么..."的具体场景描述
